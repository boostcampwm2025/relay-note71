# 퀘스트 목록

### **1. 💬 AI 추천 토론 주제 투표**

**목적:** 미션 관련한 주제 또는 자유롭게 주제를 정하여 캠퍼 간의 다양한 의견을 공유하여 학습 효과를 높이고 소통 촉진

**요구사항:**

- AI에게 **전날 미션이나 자유주제를 기반으로 난이도별(상/하)** 토론 주제 2가지를 추천받는다.
- 슬랙 `랜덤` 채널에 공유한다.
- 각 주제에 대해 이모지로 투표한다.
    - 👍 찬성 / 👎 반대
- 투표 마감 시간을 지정하고, 마감 후 결과(찬반 비율)를 정리 댓글로 공유한다.

**결과**

```
오늘의 토론 주제
1️⃣ MAC이 window보다 코딩하기 적합하다.
👍 찬성 / 👎 반대
2️⃣ 아침형 인간이 저녁형 인간보다 더 건강하다.
👍 찬성 / 👎 반대
```

---

### **2. 🖼️ AI 이미지 vs 실제 이미지 퀴즈**

**목적:** 슬랙 내 자연스러운 참여 유도, AI 이미지에 대한 감별력 및 리터러시 향상

**요구사항:**

- AI에게 생성된 이미지 혹은 본인의 실제 사진 중 하나를 선택
- 슬랙 `갤러리` 채널에 업로드하며 **"AI 생성 이미지일까요? 실제 이미지일까요?"**와 함께 투표를 유도
- 모호하고 헷갈릴만한 이미지를 선정하는 것이 핵심
- 결과는 투표 종료 후 댓글로 공개

---

### **3. 🧠 AI Gist 피드백 이벤트**

**목적:** 학습 내용을 복습하고, AI 피드백을 통해 자신의 Gist 품질 개선

**요구사항:**

- 매주 **화요일 12시**까지 자신의 **Gist URL**과 **피드백 받고 싶은 구체 항목**(예: 구조, 표현, 요약 정확도)을 제출
- AI에게 다음과 같은 프롬프트로 요청
    
    > “내 Gist(링크)에서 구조와 표현을 중심으로 개선할 점을 요약·제안해줘.”
    > 
- AI 피드백을 DM으로 수신 후, **점수(1~5)**와 **간단한 평가 이유**를 슬랙 `랜덤` 채널에 공유
- 받은 평가를 바탕으로 **프롬프트 혹은 Gist**를 개선해본 결과를 한 줄 요약해 댓글로 게시

**결과**

```
• AI 피드백 점수: 4/5
• 개선 포인트: “섹션 간 흐름을 번호 목록으로 정리하면 가독성이 더 좋아집니다.”
• 개선 후 요약: “섹션 구조를 번호 매김 방식으로 수정해 가독성 강화 완료”
```

---

### **4. 📊 어려웠던 학습 키워드 투표 및 AI 정리 공유**

**목적:** 커뮤니티 투표를 통해 공통 난관 키워드를 도출하고, AI 설명으로 이해도 향상

**요구사항:**

- 매주 **금요일 11시**에 AI에게 **이번 주 미션 키워드 5개**를 추천받는다.
    
    > “이번 주 학습한 주요 키워드 5개를 추천해줘.”
    > 
- 슬랙 `랜덤` 채널에 숫자 이모지(1~5)로 투표 진행
- 투표 마감 후 **최다 득표 키워드**를 AI에게 다음과 같이 요청
    
    > “투표 1위 ‘가상 메모리’에 대해 개념 설명과 실무 예시를 알려줘.”
    > 
- AI 설명 내용을 슬랙에 공유

**결과**

```
• 1위 키워드: 가상 메모리 (득표율 45%)
• AI 설명: “가상 메모리는 … 실제 OS 예시: Linux에서 swap 공간…”

```

    ### **1. 구조 (Structure)**
    
    ### **긍정적인 점 👏**
    
    - **훌륭한 모듈화**: `my-git.js`는 명령어 분배(router) 역할만 하고, 실제 로직은 `init.js`, `add.js` 등 각 파일로 분리한 구조는 매우 이상적입니다. `objects-helper`, `refs-helper` 등 기능을 기준으로 헬퍼 파일을 나눈 것도 코드의 재사용성과 유지보수성을 크게 높입니다.
    - **명확한 책임 분리**: 각 파일이 하나의 명령어, 하나의 기능(Single Responsibility)을 담당하도록 설계되어 코드를 이해하고 테스트하기 쉽습니다.
    
    ### **개선 제안 💡**
    
    - **README 문서 구조 개편**: 현재 `README.md`는 **학습 일지, 설계 문서, 회고**가 한데 섞여 있어 내용이 방대하고 원하는 정보를 찾기 어렵습니다.
        - **`README.md`**: 프로젝트의 최종 사용자(혹은 평가자)를 위한 문서로 만드세요. **프로젝트 소개, 주요 기능, 설치 및 사용법(`test-my-git.sh` 실행법 등)** 위주로 간결하게 작성하는 것이 좋습니다.
        - **`DEVELOPMENT.md` 또는 `docs/` 폴더**: 지금의 README 내용처럼 학습 과정, 설계 고민, 버그 해결 과정(clone 시 `process.cwd()` 문제 등)은 별도 파일로 분리하면 프로젝트의 깊이를 보여주는 훌륭한 기술 문서가 됩니다.
    - **테스트 구조**: `my-git.test.js` 파일을 통해 단위 테스트를 도입한 것은 매우 좋은 시도입니다. 다만, 회고에 언급하신 것처럼 일부 테스트가 실패하는 문제를 해결하고, 각 명령어에 대한 테스트 케이스를 더 보강하여 커버리지를 높이면 프로젝트의 신뢰도가 크게 향상될 것입니다.
    
    ---
    
    ### **2. 표현 (Expression)**
    
    ### **긍정적인 점 👏**
    
    - **명확한 변수/함수명**: `saveObject`, `getHeadCommitHash`, `handleCommit` 등 코드의 역할을 직관적으로 알 수 있는 영어로 함수와 변수명을 잘 작성했습니다.
    - **상세한 주석 및 기록**: 단순히 코드를 작성하는 데 그치지 않고, 왜 그렇게 설계했는지, 어떤 문제를 겪고 어떻게 해결했는지 (`학습정리.md`, `README.md`) 상세히 기록하여 기술적 의사결정 과정을 명확하게 표현했습니다.
    - **실제 Git과 유사한 구현**: `Buffer`를 직접 다루어 tree 객체를 바이너리 형식으로 저장하거나, `zlib`으로 객체를 압축하는 등 실제 Git의 동작 방식을 정확하게 표현하려 노력한 점이 인상적입니다.
    
    ### **개선 제안 💡**
    
    - **사용자 피드백 통일**: `console.log`, `console.error`를 사용한 사용자 피드백이 각 파일에 흩어져 있습니다. 출력 메시지의 형식(예: `[오류]`, `[성공]`)을 통일하거나, 색상을 일관되게 사용하면 더 완성도 높은 CLI 애플리케이션처럼 보일 것입니다.
    - **하드코딩된 값 분리**: `commit.js`에 작성자 정보(`"Your Name <your.email@example.com>"`)가 하드코딩 되어있습니다. 실제 Git처럼 설정 파일(`my-git-config`)을 읽어오거나, 환경 변수에서 가져오는 방식으로 개선하면 더 유연한 구조가 됩니다.
    - **회고록과 공식 문서 톤 분리**: README에 작성된 "하하...?", "...인 것 같다." 와 같은 표현은 학습 과정을 보여주는 좋은 기록이지만, 최종 사용자용 `README.md`에서는 조금 더 간결하고 객관적인 톤으로 다듬는 것을 추천합니다.
    
    ---
    
    ### **핵심 개선 제안 요약**
    
    1. **README 재구성 📑**: 사용자를 위한 `README.md`와 개발 과정을 기록하는 `DEVELOPMENT.md`로 분리하여 가독성을 높이세요.
    2. **테스트 완성 ✅**: 실패하는 테스트를 해결하고, 테스트 케이스를 보강하여 프로그램의 안정성을 증명하세요.
    3. **로직 심화 (자체 목표)**: 회고에 남기신 것처럼, `add`나 `commit` 시 하위 디렉토리까지 재귀적으로 처리하여 tree 객체를 만드는 후위 순회 로직을 구현하면 Git의 핵심을 더 깊이 이해하는 경험이 될 것입니다.
    4. **설정 외부화 ⚙️**: 작성자 정보 등 하드코딩된 값을 별도의 설정 파일로 분리하여 유연성을 높여보세요.
